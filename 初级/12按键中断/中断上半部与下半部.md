## 1. 中断上半部与下半部

### 上半部 (Top Half)

- **核心任务**：快速响应硬件。读取中断状态寄存器、清除中断标志位、收发对时间极度敏感的数据。
- **运行状态**：在**中断上下文**中运行，通常会屏蔽当前中断（甚至全局中断）。
- **限制**：代码必须精简，**严禁睡眠**。如果上半部运行太久，会导致系统失去响应或丢失其他中断。

### 下半部 (Bottom Half)

- **核心任务**：处理剩余的耗时工作（如数据拷贝、复杂的协议解析等）。
- **运行状态**：在上半部执行完并退出中断上下文后，由内核在合适的时机调度执行。
- **特点**：运行在非中断环境下（Tasklet 在软件中断上下文，Workqueue 在进程上下文）。

------

## 2. 下半部的两种主流机制：Tasklet vs Workqueue

### Tasklet (小任务)

Tasklet 是基于 **软中断 (Softirq)** 实现的。它适用于那些执行时间较短、但又不适合放在上半部的逻辑。

- **运行环境**：**原子上下文（软件中断上下文）**。
- **限制**：**不能睡眠**，不能阻塞，不能访问用户空间数据。
- **并发性**：同一类型的 Tasklet 在同个时刻只能在一个 CPU 上运行。
- **优先级**：优先级高于普通进程，执行非常快。

### Workqueue (工作队列)

工作队列将工作推迟给一个**内核线程**去执行。

- **运行环境**：**进程上下文**。
- **优势**：**可以睡眠**，可以调用可能阻塞的函数（如 `msleep()`、信号量获取、大内存分配等）。
- **并发性**：可以被内核调度器抢占。
- **开销**：由于涉及进程切换，上下文切换的开销比 Tasklet 大。

------

## 3. 核心区别对比表

| **特性**              | **Tasklet**                              | **Workqueue (工作队列)**                 |
| --------------------- | ---------------------------------------- | ---------------------------------------- |
| **执行上下文**        | 软件中断上下文 (Atomic)                  | 进程上下文                               |
| **是否允许睡眠/阻塞** | **否**                                   | **是**                                   |
| **实现原理**          | 基于 Softirq                             | 基于内核线程 (Worker Thread)             |
| **调度开销**          | 极小 (直接在中断返回前或 ksoftirqd 执行) | 较大 (涉及进程切换)                      |
| **典型用途**          | 快速处理、不需要睡眠的任务               | 需要长时间处理、可能阻塞的任务 (I/O, 锁) |
| **内存访问**          | 无法访问用户空间                         | 可以通过特定方式访问用户空间             |

------

## 4. 如何选择？

根据您的需求，可以参考以下准则：

1. **必须要睡眠吗？** 如果代码里有 `mutex_lock`、`msleep` 或磁盘 I/O，**必须用 Workqueue**。
2. **对实时性要求极高吗？** 如果需要尽快处理且任务很短，Tasklet 更合适，因为它没有进程调度的延迟。
3. **任务很重吗？** 如果任务涉及大量循环或耗时计算，建议使用 Workqueue，以免 Tasklet 长时间霸占 CPU 导致系统“卡顿”。





## 1. Tasklet 的用法详解

Tasklet 运行在软中断上下文，适用于执行速度快、不阻塞的任务。

```c
struct tasklet_struct
{
    struct tasklet_struct *next;      /* 链表指针，指向下一个 tasklet */
    unsigned long state;              /* tasklet 状态：TASKLET_STATE_SCHED 或 TASKLET_STATE_RUN */
    atomic_t count;                   /* 引用计数：0 表示激活，非 0 表示挂起/禁止 */
    void (*func)(unsigned long);      /* 核心：下半部回调函数 */
    unsigned long data;               /* 核心：传递给回调函数的参数 */
};
```

### 核心步骤：

1. **定义 Tasklet 结构体**：使用 `struct tasklet_struct`。

2. **编写回调函数**：

    - 原型：`void tasklet_handler(unsigned long data);`（注意：旧版本内核）
    - **新版本内核 (5.x+)** 改为：`void tasklet_handler(struct tasklet_struct *t);`

3. **初始化**：

    - **静态**：使用 `DECLARE_TASKLET()` 宏。

        ```c
        DECLARE_TASKLET(name, func, data);
        /*
        定义变量并初始化
        参数：name:中断底半部tasklet的名称
        	 Func:中断底半部处理函数的名字
        	 data:给中断底半部处理函数传递的参数
        */
        ```

    - **动态**：使用 `tasklet_setup()` 函数（取代了旧的 `tasklet_init(struct tasklet_struct *t, void (*func)(unsigned long), unsigned long data);`）。

        ```c
        void tasklet_init(struct tasklet_struct *t,void (*func)(unsigned long), unsigned long data)
        ```

4. **调度执行**：在中断上半部中调用 `tasklet_schedule(struct tasklet_struct *t);`。

    ```c
    void tasklet_schedule(struct tasklet_struct *t)
    //参数:t:tasklet的结构体
    ```

5. **销毁**：在驱动卸载时调用 `tasklet_kill()`。

## 2. Workqueue 的用法详解

Workqueue 运行在进程上下文，允许睡眠、阻塞和处理繁重任务。

```c
struct work_struct {
    atomic_long_t data;               /* 包含状态位和挂载的队列信息 */
    struct list_head entry;           /* 用于挂载到工作队列的链表节点 */
    work_func_t func;                 /* 核心：回调函数指针 */
};

/* 老版本回调函数原型 */
typedef void (*work_func_t)(struct work_struct *work);
```



### 核心步骤：

1. **定义工作结构体**：使用 `struct work_struct`。
2. **编写处理函数**：
    - 原型：`void work_handler(struct work_struct *work);`
3. **初始化**：
    - **静态**：`DECLARE_WORK()`。
    - **动态**：`INIT_WORK()`。
4. **调度执行**：
    - 使用系统默认队列：`schedule_work()`。
    - 使用自定义队列：`queue_work()`（适用于需要精细控制并发或优先级的场景）。
5. **退出处理**：调用 `cancel_work_sync()` 确保任务已停止。
